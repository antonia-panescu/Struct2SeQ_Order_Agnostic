Code citations:

- `run.py:407–467` (`play()`) is used for *both* train-data collection
  (`save_data=True`, `p>0`) and test play (`save_data=False`, `p=0`).
- `run.py:450`:
  ```python
  perm = torch.stack([torch.randperm(L_, device=src.device) for _ in range(B_)])
  ```
  Fresh `torch.randperm` per sample, every batch. Same code path for
  train and test play.
- `Functions.py:355` (`generate_permuted`) does the actual decode:
  - Feeds `paired_step = paired_at_pos.gather(1, perm)` — the
    forward-looking paired/unpaired indicator for the position *about
    to be predicted at step t*.
  - Passes `perm=perm` to `model.decoder(...)`, which keys the
    relative-position attention bias on it.
  - Writes outputs back to RNA-position space at the end of each step
    via `rna_outputs.scatter_(1, cur_rna_pos.unsqueeze(1), next_tokens)`.
- The legacy L→R `generate_sequence_batched` (Functions.py:490+) is
  **not** called from `play()` — kept around only for old codepaths.

For **test play** specifically: `mode="epsilon_argmax"` with `p=0` (pure
argmax), but the order itself is freshly random per sample. So our
reported `test_reward` is "argmax decode under one fresh random
permutation per target". Single-rollout — we don't yet take best-of-N
across orders. The best-of-N eval is still on the TODO list.

## Evening update — paper read + design discussions

### Read Shujun's Struct2SeQ writeup (`Struct2SeQ/writeup.pdf`)

Key findings, biggest first:

- **The §4 Discussion last paragraph names this project as future
  work.** Direct quote: *"To do this, we simply need to condition the
  model on the incomplete decoding sequence by randomly permuting
  the decoding sequence during training while applying the causal
  mask. During inference, arbitrary decoding orders can be applied."*
  That is exactly what `struct2seq_bidir_rl` implements. Reframes
  the work as "Struct2SeQ + order-agnostic extension" rather than a
  competing model. He even names two specific order strategies he
  hypothesised would help — paired-positions-together and
  loop-outward-expansion — that we can directly ablate.
- **The "3 strategies" are Beam / Random / Rescue (§2.7), not 3
  sampling temperatures.** Random and Rescue translate trivially to
  random-order. Beam over (order × token) joint space is hard; just
  pin one perm and beam-search Q-values along it.
- **Compute budget is 98 000 seqs/target, not 96.** I'd misread the
  number. They used 256 A100s × 15 h ≈ 3 840 A100-hr for the 240mer
  challenge. Out of reach on our 4 GPUs without negotiating either
  a smaller per-target budget or external compute.
- **OpenKnot screening pipeline is open-source**:
  `https://github.com/eternagame/OpenKnotScorePipeline`. Means we
  can run the exact same screening (Jaccard + target-OK-score) on
  our model's outputs without re-running anything experimental.
- **Their rescue strategy is a degenerate in-painting** — enumerate
  valid combinations at 1–2 mismatch positions and pick the best.
  Order-agnostic generalises this to arbitrary masking ratios — that
  becomes our concrete capability story.

### Reward design — comparison with the knitnet branch

Looked at `knitnet/src/models/rna_denoiser_rl_module.py:237–249`.
The knitnet REINFORCE training uses a **different reward** from
this repo:

| | `struct2seq_bidir_rl` (this) | `knitnet` REINFORCE |
|---|---|---|
| Reward | per-position binary partner-match | F1 of base pairs (continuous) |
| Granularity | vector `[L]` | scalar per sequence |
| Algorithm | DQN per-step TD | REINFORCE + baseline |
| Post-process | exp-reweight (paper eq. 6) | three modes: `raw` / `final_wc_repair` / `final_wc_repair_edit_penalty` |

Honest read: neither is universally better. Reward design follows
algorithm choice. F1 is graded (good for REINFORCE/GRPO advantage
normalisation) but ignores unpaired correctness entirely (an
all-unpaired design under a sparse-pair target gets F1 = 0 with no
gradient on the correctly-unpaired positions). Partner-match-binary
rewards unpaired correctness and gives per-position credit (good
for DQN) but has a binary cliff between off-by-one and on-target
plus a 3-position amplification cost per shifted pair.

A clean middle ground worth implementing: **partner-distance
reward** — for paired positions in the target, credit
`exp(-|j_design - j_target| / σ)` for the design's predicted
partner; for unpaired-target positions, keep the binary
unpaired-correctness term. Graded *and* per-position vector shape
(DQN still works). σ ≈ 3 nt as a starting hyperparameter.

### Bugs found and fixed during smoke testing

1. **GraphConv NaN explosion on unpaired-position rows.**
   `Encoder_Decoder.py:89` does `adj_matrix /
   adj_matrix.sum(-1, keepdim=True)` — divides by 0 when a row
   has all-zero entries. Training builds `ct_matrix` as `np.eye(L)`
   (Dataset.py:78) so every row has the diagonal=1, but my initial
   eval-script collate used `torch.zeros` and only set diagonals for
   paired positions. Result: encoder produced all-NaN, decoder argmax
   collapsed to index-0 everywhere, every generated sequence was
   `AAAAA...`. Fixed by `ct_matrix[b].fill_diagonal_(1.0)` on every
   row, mirroring training.
2. **bf16 instability in `generate_permuted` sampling on tight
   pseudoknots.** `torch.multinomial` would crash with
   `RuntimeError: probability tensor contains either inf, nan or
   element < 0` on OK7 puzzle 0 (Thermotoga fluoride riboswitch).
   Cause: bf16 model outputs occasionally NaN under tight constraint
   masks. Fixed in `Functions.py:469-498` by:
   (a) sanitising `mv32` with `nan_to_num` before argmax/multinomial,
   (b) skipping the multinomial path entirely when `p == 0` (test
   play uses argmax-only anyway, so the multinomial was dead code in
   that path). No effect on training behaviour (training uses `p > 0`).

Both fixes are likely also load-bearing for the in-flight test plays
on tight pseudoknot data; the training run hasn't crashed, so they
must be silently making low-quality decisions on those puzzles.
Worth a separate post-mortem after the paper.

### Principled motif in-painting (added per Antonia's review)

Random-scatter in-painting is the easy version: scattered fixed
positions act as anchors at every step. The honest synthetic-biology
use case is motif preservation — fix a *contiguous structurally-coherent
unit* and design the surrounding scaffold (RFdiffusion / ProteinMPNN
analog).

Implemented `evaluation/motif_extraction.py` (~200 LOC) — given a
dot-bracket, decomposes it into structural-element motifs:

- **hairpin** — stem of nested pairs ending in an unpaired loop
- **internal_loop** — stem with one inner stem, separated by unpaired
  flanks
- **multi_loop** — stem closing a junction of ≥2 inner stems
- **pseudoknot_stem** — contiguous run of crossing pairs

Per-puzzle motif inventory: 105 motifs across 20 OK7 puzzles
(26 hairpins, 6 internal_loops, 73 pseudoknot_stems, avg 5.2/puzzle).
Filtered to size ≤ 50% of puzzle length (skip degenerate "fix 89%
of structure" cases).

Eval mode: each of K=1000 samples per puzzle picks one motif at random
from that puzzle's inventory; positions in that motif fixed to WT;
everything else designed by the model under random-perm decoding.
Per-row CSV records `motif_kind` and `motif_size_nt`.

Results (K=1000 × 20 puzzles, motif mode, mixed kinds):

|              | n samples | mean J | mean OK | % perfect |
|---|---|---|---|---|
| hairpin        | 5 181 | 0.917 | 83.3 | **42.3%** |
| pseudoknot_stem | 14 073 | 0.888 | 81.8 | 37.0% |
| internal_loop  |   842 | 0.883 | 81.9 | 25.5% |
| **combined**   | **20 096** | **0.895** | **82.2** | **37.9%** |

By motif size — sweet spot at 16-25 nt:

| motif size | n | % perfect |
|---|---|---|
| 1-5 nt   | 1 525 | 10.1% (too small) |
| 6-10 nt  | 5 539 | 33.5% |
| 11-15 nt | 7 304 | 37.1% |
| **16-25 nt** | **5 042** | **56.8%** ← anchor sweet spot |
| 26-50 nt |   686 | 5.7% (too constrained) |

**Comparison vs random-scatter in-painting at matched mean fraction-fixed
(~30%, comparable to scatter K=0.25):**

| mode | % perfect |
|---|---|
| random scatter K=0.25 | 27.7% |
| **structural motifs (mean K~0.30)** | **37.9%** |

**Principled motif-fixing produces 37% more perfect designs than random
scatter at matched constraint level** — the structurally-coherent
constraint is *easier* for the model than scattered same-fraction
constraints, because the model gets to design coherent surrounding
context rather than fill in random gaps.

Constraint-held verification: 100% on a 500-row sample. The fixed motif
positions in every generated sequence match WT exactly.

This is the workshop paper's capability story. No published RNA inverse
folding paper has reported structural-element motif preservation; we
fill that gap.
