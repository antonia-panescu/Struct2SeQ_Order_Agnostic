## Shujun's question: random decoding orders at inference?

> **Shujun:** are you using random decoding orders during inference?

**Yes — fully random per-sample, no L→R fallback. Same as training.**

Code citations:

- `run.py:407–467` (`play()`) is used for *both* train-data collection
  (`save_data=True`, `p>0`) and test play (`save_data=False`, `p=0`).
- `run.py:450`:
  ```python
  perm = torch.stack([torch.randperm(L_, device=src.device) for _ in range(B_)])
  ```
  Fresh `torch.randperm` per sample, every batch. Same code path for
  train and test play.
- `Functions.py:355` (`generate_permuted`) does the actual decode:
  - Feeds `paired_step = paired_at_pos.gather(1, perm)` — the
    forward-looking paired/unpaired indicator for the position *about
    to be predicted at step t*.
  - Passes `perm=perm` to `model.decoder(...)`, which keys the
    relative-position attention bias on it.
  - Writes outputs back to RNA-position space at the end of each step
    via `rna_outputs.scatter_(1, cur_rna_pos.unsqueeze(1), next_tokens)`.
- The legacy L→R `generate_sequence_batched` (Functions.py:490+) is
  **not** called from `play()` — kept around only for old codepaths.

For **test play** specifically: `mode="epsilon_argmax"` with `p=0` (pure
argmax), but the order itself is freshly random per sample. So our
reported `test_reward` is "argmax decode under one fresh random
permutation per target". Single-rollout — we don't yet take best-of-N
across orders. The best-of-N eval is still on the TODO list.

## Resume plan (when GPUs free)

Verified resume args from code reading:

```
bash launch_brev.sh --from-pretrained policy_network_3.pt --start-episode 4
```

What this does:

- Loads `policy_network_3.pt` (start-of-Ep4 weights).
- `--start-episode 4` skips episodes 0–3, fast-forwards `p` and
  `train_batch_size` (run.py:797–799), and enters the loop at run-internal
  `episode=4`.
- Auto-skip-play (run.py:822) kicks in because
  `tmp/episode4/process*/data.pt` already exists.
- Mid-epoch resume (run.py:907) kicks in for epoch 1 — loads
  `tmp/train_mid_ckpt_ep4_epoch1.pt`, picks up at saved step ~13754.
- Epoch 0 of this Ep4 training will rerun fresh from `policy_network_3.pt`,
  which matches the original sequence (no over-training).

Alternative: load `policy_network_4_epoch0.pt` instead. Simpler mental
model but causes ~1 epoch of over-training: we'd start from already-
post-epoch-0 weights, then re-run epoch 0 on top before the
mid-ckpt-epoch1 file kicks in.
