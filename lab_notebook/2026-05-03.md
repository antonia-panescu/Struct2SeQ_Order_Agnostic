### 2. `torch.compile` + DDP key-prefix mismatch (`run.py:962`)

After re-enabling `torch.compile` in resume #2 (PID 3332489), the line:

```python
target_network.load_state_dict(policy_network.state_dict())
```

crashed at the end of Ep1 training (5 epochs successfully completed — all
per-epoch checkpoints saved to disk). Asymmetric key prefixes after
`compile + DDP` layering: one wrapper produced
`_orig_mod.module.encoder.*`, the other `_orig_mod.encoder.*` →
load_state_dict with strict=True crashed with Missing/Unexpected key
errors.

**Fix** (commit `834a1d5`):
```python
accelerator.unwrap_model(target_network).load_state_dict(
    accelerator.unwrap_model(policy_network).state_dict()
)
```

`accelerator.unwrap_model()` strips both the compile and the DDP wrapper.

### 3. `_orig_mod.` prefix on checkpoints (load) + `UnboundLocalError` (skip-train)

When relaunching with `--start-episode 1 --skip-train --checkpoint
policy_network_1.pt` to skip retraining:

a. **Load failed** because the per-epoch checkpoint had been saved during
   the compiled run, retaining `_orig_mod.encoder.*` key prefixes.
   `accelerator.unwrap_model()` strips DDP but **does NOT** strip torch.
   compile's `OptimizedModule` wrapper. So the on-disk checkpoint had the
   prefix even though we used `unwrap_model` at save time.

   **Fix** (commit `e7bc684`): defensive prefix-strip on load:
   ```python
   def _clean_key(k):
       for prefix in ("_orig_mod.", "module."):
           if k.startswith(prefix):
               k = k[len(prefix):]
       return k
   state = {_clean_key(k): v for k, v in state.items()}
   ```

b. **`UnboundLocalError: optimizer`** at `run.py:1024` because
   `--skip-train` skips the optimizer creation, but the metrics-write
   path unconditionally read `optimizer.param_groups[0]["lr"]` for the
   `lr_final` column.

   **Fix** (commit `49af2ce`): try/except → fall back to NaN.

## inference_batch_size 64 → 128 effect

The current Ep2 play phase is using B=128:
- 3125 batches/rank (vs 6250 with B=64) — half the iterations
- Pace: ~7.8 s/it (vs 5.7 s/it at B=64) — about 1.4× per-iter
- Net wall time: ~7 hours, slightly faster than the 9.5h with B=64 but not 2×
- The RibonanzaNet folder remains the bottleneck — it scales nearly linearly
  with batch in time, so the doubling of batch only cuts wall time by maybe
  20–30% rather than 50%.

So the play-phase wins are smaller than I'd hoped from the bench. Future
options to attack this: (a) compile the env.SS_model differently (it's
already compiled in Env.py:101–103 but still slow), (b) explore lower-cost
folders (e.g., a smaller model trained for SS only), (c) RoPE for flash V2
on policy decoder (won't help fold but free up more time).
