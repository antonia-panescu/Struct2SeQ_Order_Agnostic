## Goal

## What happened

### Crash: NCCL timeout at end of play phase
Both the `knitnet_rl` session (launched Apr 19) and a subsequent attempt crashed with `SIGABRT` / `c10::DistBackendError` at the end of the play phase.

**Root cause:** `play()` in `run.py` accumulated all batch structure tensors throughout the entire play loop, then called `accelerator.gather(torch.cat(episode_structures))` at the very end — attempting to all-gather potentially millions of sequence tokens across 8 GPUs simultaneously. This hit NCCL's 600s watchdog timeout every time.

**Fix applied:**
- Removed `episode_structures` accumulation from `play()` entirely.
- `play()` now only gathers reward scalars (cheap), not structure tensors.
- `play()` now returns `(mean_reward, rewards_vector)` — 2 values instead of 3.
- Removed dependent `hard_structures` curriculum logic (was partially commented out anyway).
- CSV writes for stats guarded with `accelerator.is_main_process` to prevent all-rank race condition.
- `launch_brev.sh` updated to forward unknown args to `run.py` via `$EXTRA_ARGS`.

### Resume without restarting
The crashed session had completed 89% of the episode 0 play phase (1,422,592 pkls written to `tmp/episode0/`). Added `--skip-play` flag to `run.py` to skip play for episode 0 when pkls already exist, jumping straight to training.

**Launched in `knitnet_rl` tmux session:**

```
bash launch_brev.sh --from-pretrained ~/struct2seq_bidir_rl/Struct2SeQ.pt --skip-play
```

## GPU efficiency fixes

### Training dataloader not distributed (critical bug)
`accelerator.prepare(train_dataloader)` was commented out — all 8 GPUs were training on the full dataset independently instead of sharding. Uncommented. This alone makes training ~8x faster.

### Batch sizes too small for 80 GB A100s
Original config used `train_batch_size=1`, `max_train_batch_size=64`, `inference_batch_size=128`. With fine-tuning (not scratch), there's no reason to start small. Updated to:
- `train_batch_size=256`, `max_train_batch_size=256` (fixed, no doubling — fine-tuning starts stable)
- `inference_batch_size=128` (tried 512 but OOMed during play — both policy_network and SS_model run simultaneously, ~56 GB allocation)

### Disk I/O bottleneck: 1.4M small pkl files
Initial training at batch=256 was 80 s/it due to disk reads. Fix: load all pkls into CPU RAM at `RNADataset.__init__()` (1.4M files × ~6.5 KB = ~9 GB — fine with 1.7 TB RAM on this node).

### OOM crash: precomputing ct_matrices in RAM
Attempted to precompute 240×240 float32 ct_matrices for all items at load time. This required 230 KB × 1.4M × 8 ranks = ~2.6 TB RAM — exceeded available memory, OOM killer destroyed the tmux session. Reverted to lazy ct_matrix computation in `__getitem__`, but vectorized with numpy (`np.eye` + index assignment) instead of Python loops.

### num_workers tuning
- `num_workers=16` with in-RAM data: high IPC overhead, GPUs starved (0% utilization)
- `num_workers=0`: no prefetch, GPUs still starved
- `num_workers=4, prefetch_factor=2`: 4 workers compute ct_matrices ahead of GPU — currently testing

### Current data pipeline (end of day)
1. Load pkls into CPU RAM once per episode (~1-2 min, 9 GB/rank)
2. DataLoader workers (×4) compute ct_matrix per item on CPU, prefetch ahead
3. GPU trains on batches of 256

### Remaining open question
If 4 workers still can't keep GPUs fed, next step is to precompute and store **bp lists** (sparse, ~few hundred bytes each) during the load step instead of recomputing from dot-bracket strings each time. This avoids the RAM explosion of dense matrices while eliminating the expensive string decode + `convert_dotbracket_to_bp_list` call per item.

## Original paper hardware context (Struct2SeQ, Shujun He et al., bioRxiv 2026.01.16.700031)

- **100mer training**: 8× A6000 GPUs, ~15 hours
- **240mer training**: 256× A100 GPUs (32 nodes × 8), ~15 hours, ~4,000 A100 GPU-hours
- **inference_batch_size=128** in the original config was designed for 256 GPUs → effective per-GPU batch ≈ 0.5–1 item
- On our 8× A100 setup, inference_batch_size=128 gives ~16 items/GPU → 32× more memory pressure per GPU on the SS_model outer product `[B, L, L, C, D]`
- This is why test play OOM'd (tried to allocate 14 GiB) after training, when optimizer states were also resident in GPU memory
- **Fix applied (Apr 21)**: reduced `inference_batch_size: 128 → 64` (8 items/GPU) — still OOM'd during test play after episode 0 training (got 68% through at batch_size=64 before SIGABRT on rank 7)
- **Fix applied (Apr 22)**: reduced further to `inference_batch_size: 32` (4 items/GPU) — at half the memory of 64, should fit comfortably; if it OOMs again, next step is 16
- Root cause: optimizer states (Ranger first+second moments, ~3× model size) stay in GPU memory during test play, leaving less headroom than during the initial training play phase; test play OOM is an episode 0 problem that repeats every episode
- The original researchers never hit this OOM because per-GPU batch was effectively ~1 on their 256-GPU cluster
- **Side effect**: halving batch_size doubled play phase duration — at batch_size=32, episode play phase takes ~12h (6,250 batches) vs ~6h at 64

## Deferred optimizations (NOT yet applied — review before next long run)

These were tested during the Apr 20 session but reverted to preserve original training logic. Re-evaluate once baseline RL run completes.

### 1. Distribute the training dataloader (DDP fix)
**What:** Uncomment `train_dataloader = accelerator.prepare(train_dataloader)` as a standalone call *before* the `accelerator.prepare(scheduler, optimizer, train_dataloader)` tuple call — or equivalently, ensure the dataloader shard is applied so each GPU trains on `1/8` of the data per step rather than all of it.
**Why:** With 8-GPU DDP, gradients are averaged across ranks but each rank currently processes the entire dataset — effectively 8× redundant compute. Distributing the dataloader would give a true ~8× throughput gain.
**Risk:** This changes training dynamics (effective batch size per update changes). Validate that loss converges similarly before a full run.

### 2. Increase train batch size
**What:** Set `train_batch_size: 256`, `max_train_batch_size: 256` in `config_brev_8gpu.yaml` (or similar large fixed value — no episode doubling).
**Why:** A100-SXM4-80GB GPUs have plenty of headroom. Batch size 1 keeps GPU utilization extremely low (measured ~80 s/it vs potentially ~5 s/it). Fine-tuning starts stable so the original warmup-from-batch-1 ramp is unnecessary.
**Risk:** Disables the original curriculum doubling schedule. Start from a small value (e.g. 64 or 128) if concerned about early-episode instability.

### 3. Precompute bp lists during dataset load
**What:** In `RNADataset.__init__`, call `convert_dotbracket_to_bp_list` for each structure once at load time and cache as `self.bp_lists`. `__getitem__` then builds ct_matrix from the cached list instead of re-decoding the dot-bracket string.
**Why:** `detokenize_dot_bracket` + `convert_dotbracket_to_bp_list` is called on every item every epoch. Caching eliminates this CPU bottleneck without the RAM explosion of precomputing dense ct_matrices.
**Risk:** Negligible — bp lists are sparse (~few hundred bytes each). Total ~1.4M × ~300 bytes = ~420 MB per rank, well within budget.
