## Speed reality vs. earlier overstatement

I had claimed 5–10× speedup earlier; corrected today with actual numbers.

**Old-arch reference** (re-derived from `logs/metrics_step.csv` window
2026-04-27 with B=256, 4-GPU DDP): 100 steps spanned 65.4 s →
**391.6 samples/sec total = 97.9 samples/sec/GPU** (my earlier "19.8"
reference was wrong — the audit-agent had mis-derived it).

**New-arch training-step benchmark** on a single A100-80GB at L=240
(`evaluation/bench_train_speed.py` — added today, committed):

| B | K | effective | samp/s/GPU | Mem GB | Speedup vs OLD |
|---|---|---|---|---|---|
| 256 | 1 | 256 | **601** | 24 | 6.1× |
| 512 | 1 | 512 | 624 | 48 | 6.4× |
| 128 | 4 | 512 | **612** | 48 | 6.3× |
| 64 | 8 | 512 | 613 | 48 | 6.3× |
| 256 | 4 | 1024 | OOM | — | — |

Throughput plateaus around **effective batch 256–512** — beyond that
we're memory-limited, not compute-limited. K=4 vs K=1 at the same
effective batch gives the same throughput, but K=4 gives Shujun's
permutation-invariance signal (4 different orders on the same target
per gradient step).

Headline: **~6× faster training**, not 30× and not 10×. The earlier
overstatement was because the OLD baseline reference number (19.8) was
wrong; corrected to 98 → ~6× speedup.

**Why training 6× but eval only ~2×?** Different bottlenecks:
- Training: ONE forward pass per step at length L. Removing the LSTM/Conv
  layer is a single-layer compute saving. Modest constant-factor win.
- Inference (autoregressive): L decoding steps. The OLD LSTM/Conv
  *recomputed on the FULL grown tgt every step* — total O(L²) extra
  work per sample. New arch removes that → big win on the policy decode.
  But the **fold step (RibonanzaNet)** is unchanged and is ~50% of eval
  wall time, capping the total speedup to ~2×.

## Flash attention status (it's not, decoder)

Not currently using flash attention in the decoder. The new RPE design
adds an additive logit-space bias (per-head `[B*nhead, Lq, Lk]`); PyTorch
2.3's flash V2 backend rejects additive masks and falls back to
**memory-efficient SDPA**. We do get flash on the encoder
(`nn.TransformerEncoderLayer` with no per-head bias).

The `torch.backends.cuda.sdp_kernel(enable_flash=True,
enable_mem_efficient=True, enable_math=False)` hint at startup disables
the math kernel so PyTorch uses mem-efficient when flash isn't eligible.

Future: switching to RoPE-style PE applied to Q/K (instead of additive
bias) would unlock flash V2 on the decoder — possible follow-up.

## On Shujun's "PE[random_order, random_order]" suggestion

For the record: the additive-RPE-bias design we shipped IS what Shujun
recommended. He wrote:

> *"For order-agnostic decoding, you can just use PE[random_order,
> random_order], where PE is LxLxC tensor that encodes relative position.
> Originally i just used LSTM+causal conv because it was easy to code up."*

What we have in `Encoder_Decoder.py:RelativePositionBias` is exactly
this: a per-head bias table indexed by `bucket(perm[i] - perm[j])`, added
to attention scores via `attn_mask`. The only twist vs Shujun's "LxLxC
tensor" is T5-style bucketing — 64 buckets × n_heads ≈ 800 parameters
total instead of the dense `Lmax² × n_heads` (~3M params for Lmax=512).
Functionally the same; we save parameters and generalize past
training-time sequence lengths.

This is **NOT the same as RoPE** (which on the future-work list is for a
different reason — flash V2 compatibility, see below).

## Speedup track 3 (queued for next launch)

Three follow-on speedups, all of which apply only to a NEW process start
(not to the running Ep1) — i.e. they take effect on the next preemption +
resume, or after Ep1 finishes manually. **Local-only commits, not pushed**:

1. **`inference_batch_size: 64 → 128`** (commit `04dc9bf`). Play phase is
   ~50% of episode wall time and B=64 only used ~22-34 GB / 80 GB. B=128
   should give 1.5-2× faster play. Memory pressure on long-tail
   structures (L>~350) might OOM at chunk-of-128; the incremental save
   handles that.

2. **Re-enable `torch.compile`**. The current run was launched with
   `STRUCT2SEQ_DISABLE_COMPILE=1` for first-run safety, but `run.py:766`
   enables compile by default if that env var is unset. The next
   `launch_brev.sh` invocation (which doesn't set the var) will pick up
   compile automatically — `mode='default'`, `dynamic=True`, applied to
   both policy and target networks after `accelerator.prepare`.
   Expected 1.2-1.5× training. If Ep1 of the compiled run runs cleanly,
   can escalate training-only to `mode='reduce-overhead'` (CUDA graphs,
   1.5-2× on training) since training shapes are fixed.

3. **(Future, not for this run)** RoPE-style PE applied to Q/K instead
   of additive RPE bias. Reason: PyTorch 2.3's flash V2 SDPA backend
   rejects additive `attn_mask` and falls back to mem-efficient. RoPE
   modifies Q/K *before* attention, leaving the attention call seeing
   plain Q,K,V — flash V2 eligible. Expected another 1.3-1.6× on the
   decoder attention (mostly play-phase win). Note this is a
   **different** speed lever than Shujun's bias suggestion; we'd be
   trading equivalent expressiveness for a different speedup vector.
   Would require re-running fine-tune from `Struct2SeQ.pt` since the
   weights would interact with the rotary mechanism differently.
