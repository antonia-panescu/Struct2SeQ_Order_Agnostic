## Crash: SIGABRT at ~39% of episode 1 play phase (twice)

Episode 1 play phase crashed twice with SIGABRT (signal 6, OS-level abort) at almost exactly **39% through 6,250 batches** (~10.5 hours in), on rank 3 both times. No Python traceback — the process was killed at the CUDA/OS level, not by a Python exception.

### What we know
- Crash is **deterministic** — same percentage both times, same rank (3). Suggests a specific batch of data is triggering it.
- **Zombie GPU memory** from crashed processes on GPUs 3 and 4:
  - GPU 3: ~5.2 GB stuck (PID `[Not Found]`) — cannot be freed without GPU reset
  - GPU 4: ~7.8 GB stuck (PID `[Not Found]`) — cannot be freed without GPU reset
  - This gives GPUs 3 and 4 ~5–8 GB less headroom than the other GPUs
- **Other jobs on GPU 0**: Two `enhanced_DSA.py` processes (PIDs 1925804, 2037849) running since Apr 1, using 516 MB each — not our jobs, not causing crashes
- Likely causes: (1) CUDA OOM on GPU 3 due to zombie memory reducing headroom, (2) Azure VM preemption (Brev runs on spot instances), (3) specific bad batch of data

### Fixes applied
- Added **incremental saves** to `play()` every 500 batches (`save_interval=500`) — future crashes won't lose all play data
- Auto-skip play for any episode where `data.pt` already exists (generalized crash recovery, not just episode 0)
- `--start-episode N` flag added so we can resume from any episode without redoing prior training

## Training schedule change (Apr 25)

**Episodes 0–2 (completed):** `epochs_per_episode: 1` — trained on accumulated data once per episode before collecting new data.

**Episodes 3–9 (going forward):** `epochs_per_episode: 5` — increased to match original paper's 10×5 schedule (Shujun He et al.). Rationale:
- Play phase (~12h) is the bottleneck; 1 epoch discards 12h of collected experience after a single pass
- 5 epochs extracts ~5× more gradient signal from the same data collection cost
- At current speed, 5 epochs ≈ 17.5h training vs 3.5h before; total cycle time ~29.5h vs ~15.5h
- Training on all accumulated episode data means later epochs train on increasingly rich mix of old+new experience

Change applied to `config_brev_8gpu.yaml`: `epochs_per_episode: 1 → 5`. Takes effect starting episode 3.

## 2026-04-27: Switch to 4 GPUs + all speedups (PI directive)

### Changes applied
PI requested reduction to 4 GPUs to share server with labmates (ziyu, haiwen). Took this as an opportunity to implement all pending speedups:

**`launch_brev.sh`:**
- Added `export CUDA_VISIBLE_DEVICES=0,1,2,5` — explicitly restricts to GPUs 0,1,2,5 (avoiding GPUs 3,4 which have zombie memory from prior crashes, and GPU 7 which has another large job)
- Changed `--num_processes 8 → 4`

**`config_brev_8gpu.yaml`:**
- `learning_rate: 0.001 → 0.0001` (matches Shujun He's original design; we were 10× too high before)
- `train_batch_size: 1 → 256` (per GPU, global batch = 4 × 256 = 1024; see batch size iteration below)
- `max_train_batch_size: 256`
- `inference_batch_size: 32 → 64` (play phase; safe with SS_model now in bf16)

**`run.py` (play phase):**
- Moved `bpps = env.SS_model(predicted_sequences).sigmoid()` inside `with accelerator.autocast():` block — SS_model was running in fp32, now runs in bf16, saving ~half the model's memory during play

**`run.py` (training, bugfix):**
- Fixed `target_network.load_state_dict(unwrapped.state_dict())` at two sites (per-epoch and mid-epoch checkpoint loading) — `target_network` is DDP-wrapped (keys have `module.` prefix) but was being loaded from an unwrapped state dict (no prefix). Fixed both to `accelerator.unwrap_model(target_network).load_state_dict(...)`. This bug would have crashed any run that tried to resume from a checkpoint.

### Batch size iteration (Apr 27)

Started at batch=64, then increased to batch=256 after observing only 14% GPU memory utilization.

Memory scaling was empirically linear with batch:
- batch=8 (old, 8 GPUs): 3.8 GB/GPU
- batch=64 (first attempt): 11 GB/GPU (14%)
- batch=256 (final): **37 GB/GPU (46%)** — estimated from `(11 - 3.8)/56 × 256 + 3.8 ≈ 36 GB`, confirmed empirically

### Result (final: batch=256, 4 GPUs)
- Stopped episode 3 training (was partway through epoch 0; mid-epoch checkpoint at step 5000 saved → resumed from there on restart)
- Episode 3 play data safe (all 8 process data.pt files from prior 8-GPU run, auto-skip triggers correctly)
- Running: `bash launch_brev.sh --from-pretrained policy_network_2.pt --start-episode 3`
- Speed: **1.94 it/s @ 5,343 steps/epoch = ~46 min/epoch × 5 epochs = ~3.8h** for episode 3 training
- GPU compute utilization: **99% on 3/4 GPUs, 81% on 1** (DDP coordinator rank)
