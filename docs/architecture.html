Struct2SeQ-bidir — architecture explainer

Plain-English walk-through of every component in the bidirectional / order-agnostic RL extension of Struct2SeQ. Each card explains what a block does, why it’s needed, and where to find it in the code.

RNA inverse folding · Q-learning · order-agnostic decoding · repo figure (PDF)

TL;DR

The original Struct2SeQ decoded RNA sequences left to right using an LSTM positional encoder plus a CausalConv1d — both hard-coded that order. We replaced those with T5-style relative-position attention biases in the decoder so it becomes order-equivariant, then trained it with random per-sample decoding permutations tiled K times per target. The reward (per-position binary partner-match) is gathered through the same permutation, so the Q-learning TD update is consistent with the shuffled decode order.

Result at K=1000 on 240-mer targets: 2.32× over the L→R baseline.

What each block does

NEW Random permutation perm[B, L]
›

A random ordering of the L positions of the RNA, one per sample in the batch. It says “decode position 47 first, then position 12, then position 199 …”. The original model always decoded in order 1, 2, 3, …, L.

Where in the code

run.py:225–233 (training) and run.py:450 (generation). 

# Per-sample [B, L] random permutations: perm[b, t] is the RNA
# position predicted at step t for batch element b.
perm = torch.stack(
    [torch.randperm(L, device=sequence.device) for _ in range(B)]
)

Why we need it

A model that can decode in any order has to learn nucleotide identities that hold regardless of when each position gets filled in. The fixed L→R order in the original Struct2SeQ lets the model rely on left-context shortcuts (“at step t the model has already seen positions 0…t-1”), which is brittle and limits how the model can be combined with inference-time strategies (e.g. in-painting fixed motifs).

NEW K-permutation tiling (training only)
›

We copy each target K times in the batch and pair each copy with a different random perm. The model sees the same structure decoded in K different orders simultaneously, so it can’t get away with memorizing “what to put at step 0” — it has to learn what nucleotide belongs at each RNA position regardless of when it gets to decode it.

Where in the code

run.py:212–231; controlled by k_perm in config_brev_8gpu.yaml (default k_perm: 4). 

# K-permutation tiling: replicate each sample K times along batch dim,
# draw an independent random permutation per replica. Forces the
# decoder to learn permutation-invariant features per data sample.
K = max(1, getattr(config, "k_perm", 1))

Why we need it

Without tiling, each target sees one random order per epoch and the gradient signal for “this position should be a C” is averaged over many different decoding contexts only across the dataset. With K replicas of the same target in one batch, the averaging happens per–example per–step and the permutation-invariance constraint is enforced more tightly.

NEW Shifted-right tgt + paired_encoding gathered through perm
›

The decoder needs to know what nucleotide it picked last step, and whether the position being predicted is paired or unpaired. Once we shuffle the decoding order, we have to reorder the ground-truth sequence and the paired/unpaired indicator the same way (that’s what gather(1, perm) does). Without this, the inputs and labels would no longer line up.

Where in the code

run.py:232–245

# Permute sequence and reward to match decoding order
sequence_perm = sequence.gather(1, perm)
reward_perm = reward.gather(1, perm)
paired_encoding_perm = (structure == 0).long().gather(1, perm)

# Build shifted sequence in permuted order (START token at slot 0)
shifted[:, 1:] = sequence_perm[:, :-1]

Why we need it

Otherwise the decoder at step t would receive the previous nucleotide in the original RNA-order frame while it is being asked to predict perm[t] — a misalignment that destroys the teacher-forcing signal.

NEW RPE_self — relative-position bias for decoder self-attention ★ key architectural change
›

A learned bias added to attention scores between two decoder positions. Crucially, the bias depends on the RNA-position distance between them — perm[t1] − perm[t2] — not on how many decoder steps apart they are. Bucketed T5-style so it can express both “right next to each other” and “very far away on the strand” smoothly. This lets the decoder use spatial information on the RNA without caring about decode order.

Where in the code

Encoder_Decoder.py:196–260 (class) and :339–346 (usage)

# Decide which step indices the current tgt corresponds to.
self_q_pos = perm[:, :L]   # RNA position of each query step
self_k_pos = perm[:, :L]   # RNA position of each key step

# Self-attention bias: [B, n_heads, Lq, Lk]
self_bias = self.rpe_self(self_q_pos, self_k_pos)

Why we need it

Without RPE, the decoder gets no position information under random-order decoding (we deliberately stripped out the LSTM and CausalConv1d that used to encode it; see the “Removed” card below). RPE supplies that information in a form that is invariant to step order — the model can ask “how far is position 47 from position 199 on the RNA?” and get a consistent answer no matter when either was decoded.

NEW RPE_cross — relative-position bias for decoder → encoder cross-attention ★ key architectural change
›

Same trick, but for when the decoder looks at the encoder’s structure representation. The bias is RNA-distance between the position being predicted (perm[t]) and the encoder memory slot being read (which sits at its true RNA position, 0 … L−1). This keeps the encoder and decoder talking in the same coordinate system: RNA-space.

Where in the code

Encoder_Decoder.py:286–295 (separate parameter table) and :349–353 (usage)

# Cross-attention bias: encoder memory keys are at RNA positions 0..L_enc-1
if encoder_perm is None:
    encoder_perm = torch.arange(L_enc, device=tgt.device) \
                        .unsqueeze(0).expand(B, L_enc)
cross_bias = self.rpe_cross(self_q_pos, encoder_perm)

Why two separate RPE tables?

The distribution of relative offsets is different for self-attn (decoder ↔ decoder, both at permuted positions) vs cross-attn (decoder query at perm[t], encoder key at fixed RNA position). Sharing parameters would force one table to model both regimes; we keep them separate.

NEW Q-learning TD target with reward gathered through perm
›

Same Q-learning loss as the original Struct2SeQ — predict expected future per-position reward. The only change is that the reward at step t is the reward for the position perm[t] (not position t), so the temporal-difference target is consistent with the shuffled order. The model gets the right credit at the right step.

Where in the code

run.py:295 (TD target); run.py:233 (reward gather)

reward_perm = reward.gather(1, perm)
# …
expected_q = reward_perm + (gamma * next_q)

Why we need it

It’s mechanical, but easy to get wrong: if the reward stays in RNA order while everything else is in permuted order, the model learns to attribute step t’s loss to the wrong nucleotide. Gathering rewards through perm is what makes the RL objective compatible with permutation decoding.

REMOVED LSTM positional encoder + CausalConv1d (in the decoder)
›

The original Struct2SeQ decoder ran a one-direction LSTM and a CausalConv1d over the token embeddings to provide position information. Both are inherently left-to-right operators:

  • The LSTM is recurrent forward in time.
  • CausalConv1d only looks at past timesteps.
Under a random decode order, “the start” is meaningless — what is step 0 for one sample is step 87 for another. These layers would produce nonsense position signals. We replaced them with RPE_self / RPE_cross, which work in RNA-position space and are invariant to step order.

Where in the code

Originally in Decoder class (still present at Encoder_Decoder.py:154 for reference). Replaced by OptimizedTransformerDecoder at Encoder_Decoder.py:263–395, used by the model in DotBracketRNATransformer.

Why this matters

Removing these is what forces RPE_self and RPE_cross to do all the position work. The two changes — ripping out the L→R-baked operators and adding the order-equivariant RPE — are a single architectural move dressed as two commits.

INHERITED Encoder (unchanged)
›

The encoder still runs LSTM positional encoder → Conv1d (k=5) + Dropout → LayerNorm with residual → GraphConv (GeM pooling over the CT base-pair matrix) → N standard TransformerEncoderLayers. We didn’t touch it because the input structure (dot-bracket) is fixed and has a fixed canonical order, so the L→R operators do no harm here.

Where in the code

Encoder_Decoder.py:100–142 (class Encoder)

Why these pieces only make sense together

None of the new components is independently sensible. The story is a single methodological choice with consequences:

  1. The training-procedure changes — perm, K-permutation tiling, gathered rewards — require an architecture that can handle different decode orders without confusion.
  2. The architecture is order-equivariant only after the LSTM and CausalConv1d are removed.
  3. Removing them leaves the decoder with no position information at all. That’s why RPE_self and RPE_cross have to be added.
  4. Once decoding is permuted, the per-position reward signal has to be permuted too, otherwise the Q-learning TD update is incoherent.

Pull any one out and the rest stop working. That coupling is the point.

Is this novel or borrowed?

Honest self-assessment: the individual ingredients are not novel. The combination is, and so is the result for RNA design.

The closest cousin in the literature is ProteinMPNN. Read it carefully before tomorrow — it is doing the same conceptual move (random-order autoregressive decoding for inverse folding) for proteins, under supervised cross-entropy. Two differences that buy us novelty:

  • RNA, not protein. Different oracle, different evaluation, different structural constraints (paired/unpaired indicator, base-pair graph).
  • RL with a non-differentiable structural reward, not cross-entropy. The non-trivial coupling (gathering rewards through perm for the TD target) only exists in the RL setting.

How to pitch this at ICML

Don’t pitch this as “we invented permutation decoding”. A reviewer who has read XLNet or ProteinMPNN will close the tab.

ProteinMPNN showed order-agnostic decoding works for protein inverse folding under cross-entropy supervision. We ask: does it also work for RNA, under RL with non-differentiable structural rewards? The non-trivial part is the architecture — the original Struct2SeQ decoder bakes L→R into its LSTM positional encoder — so we replace it with T5-style relative-position bias to make the decoder permutation-equivariant. The result is the first order-agnostic RL framework for RNA design, with [headline number] improvement over the L→R Struct2SeQ baseline. — suggested framing
